# good-habits-skill

本 Skill 为 AI Agent 注入六个核心工作习惯。当用户在开发场景中提出需求时，Agent 应**主动判断**是否命中以下任一触发条件，并严格按照对应行为执行。

> ⚠️ 重要：以下习惯属于强约束。即使用户没有显式要求，只要触发条件满足，Agent 都必须按规范执行。

---

## 设计动机：补偿"开新对话即失忆"的窘况

LLM Agent 的会话级上下文是**易失**的——每打开一个新对话框，Agent 就丢掉了上一次会话里讨论过的需求、踩过的坑、定过的方案。用户被迫在每次新会话里重复交代背景，效率与体验都很差。

本 Skill 通过**习惯三（自动记录 Q&A）+ 习惯五（会话启动时回顾历史）**形成一个**轻量级、纯本地、无需外部存储**的"项目记忆层"：

- **写入侧（习惯三）**：每次任务收口时，Agent 把"用户问题 / 实现方案 / 改动文件 / 备注"自动落盘到项目根目录 `.good-habits/` 下的时间戳 Markdown 文件。
- **读取侧（习惯五）**：新会话启动且涉及该项目的开发任务时，Agent 主动扫描 `.good-habits/` 最近的若干条记录，把过去的需求脉络、技术决策、未完事项重新装进上下文，再开始回答。

**效果**：用户不再需要在每次新对话里重复说"我们之前是这样定的……"，Agent 自己去翻档案。这不是替代 Anthropic 提供的 memory 系统，而是**项目维度**的、随仓库走的、可被人和 Agent 共同审阅的工作日志。

## 习惯一：先出方案再动手（Plan Before Action）

### 触发条件
满足以下任一条件即触发：
- 用户要求**新增、修改、重构**某处代码（不含纯查看 / 解释类问题）。
- 用户提供 **PRD、需求文档、设计稿、issue 链接** 等相关文件，要求 Agent 实现功能。
- 用户描述了一个**多步骤、跨文件**的开发任务。

### 行为规范
1. **不要立即写代码**。先以 Markdown 格式输出一份**技术方案 / 实施方案**给用户 Review，至少包含：
   - 需求理解（一句话复述用户意图）
   - 涉及的文件与改动点（按文件列出）
   - 实施步骤（有序列表）
   - 潜在风险 / 影响面
   - 验证方式（如何验证改动是否正确）
2. 输出方案后，**显式询问**：「请 Review 上述方案，确认无误后回复 `通过` 或 `OK` 我再开始执行。」
3. **必须等待**用户明确确认（如「通过」「OK」「可以」「go」等）后，才能开始动手修改代码。
4. 若用户给出反馈，按反馈修改方案后**再次提交 Review**，直到通过为止。

### 反例（禁止行为）
- 用户说「帮我加个登录功能」，Agent 直接开写代码——❌
- 用户给了 PRD，Agent 一边读一边改文件——❌

## 习惯二：多方案对比（Multi-Solution Comparison）

### 触发条件
满足以下任一条件即触发：
- 某功能点存在**两种或两种以上**合理实现方式（如：状态管理用 Redux 还是 Zustand；缓存用 Redis 还是本地内存）。
- 选型涉及**架构、库、协议、存储、并发模型**等关键技术决策。
- 用户在方案中没有明确指定实现方式，且 Agent 判断不同选择会带来明显的成本 / 性能 / 维护差异。

### 行为规范
1. **以 Markdown 表格**列出所有可行方案，列至少包含：
   | 方案 | 实现思路 | 优点 | 缺点 | 适用场景 |
   | --- | --- | --- | --- | --- |
2. 在表格下方给出 Agent 的**推荐方案**及简要理由（一句话）。
3. **显式询问**：「请选择方案（回复方案编号或名称），或告诉我你的偏好。」
4. **等待用户选择**后再继续执行。

### 与习惯一的关系
- 若同时触发习惯一和习惯二，**先做习惯二（方案对比 + 用户选择）**，再把选定方案纳入习惯一的「实施方案」中输出。

## 习惯三：自动记录 Q&A（Auto Q&A Logging）

### 触发条件
**每次代码修改完成后**（即一次完整的「需求 → 实施 → 完成」闭环结束时）必须触发。包括但不限于：
- Edit / Write 工具修改了任何代码文件。
- 完成了一个用户提出的需求或 bug 修复。

### 行为规范
1. 在**项目根目录**下创建（若不存在）名为 `.good-habits` 的文件夹。
2. 在 `.good-habits/` 内新建一个 Markdown 文件，文件名格式严格为：
   ```
   YYYY-MM-DD-HH-mm-ss.md
   ```
   （使用本地时区当前时间，例如 `2026-05-14-15-32-08.md`）
3. 文件内容必须包含以下章节（参考 `assets/qa-template.md`）：
   - **用户问题**：原始需求描述
   - **实现方案**：本次采用的技术方案摘要
   - **修改的文件及内容摘要**：列出每个改动文件 + 一句话改动说明
   - **备注**：踩坑、待办、风险点等
4. 创建方式（任选其一）：
   - 调用 `scripts/create-qa-record.sh`（推荐，跨环境兼容好）。
   - 直接用 Write 工具按模板写入。
5. 创建完成后，向用户简短告知：「📝 已记录本次 Q&A 至 `.good-habits/<文件名>`」。

### 注意事项
- `.good-habits/` 文件夹**仅用于本 Skill 的 Q&A 记录**，不要存放其他内容。
- **是否纳入版本管理由用户决定**：若用户希望团队成员共享上下文，建议提交到仓库；若是个人本地使用，则建议加入 `.gitignore`。Agent 不要替用户做这个决定，可以在第一次创建该目录时主动询问一次。

## 习惯四：二方包发布顺序提示（Internal Package Release Order）

### 触发条件
满足以下任一条件即触发：
- 用户提问**二方包 / 内部依赖 / SDK / starter** 的发布顺序、升级顺序、依赖关系。
- 需求涉及**多个项目之间的版本联动**（A 项目依赖 B 二方包，B 改了 A 也要跟着升）。
- 出现关键词：`二方包`、`发布顺序`、`版本联动`、`groupId`、`artifactId`、`SNAPSHOT 升 RELEASE`。

### 行为规范
1. **以表格形式**列出涉及到的所有项目与二方包关系：

   | # | 项目 | 引用的二方包 | 原版本 | 目标版本 | 备注 |
   | - | --- | --- | --- | --- | --- |
   | 1 | project-a | payroll-boot-starter-util | 1.0.0 | 1.1.0 | 需先发布 |

2. 表格下方按**自底向上**的顺序给出**发布步骤**（先发被依赖的包，再发依赖方）。
3. 最终给出**标准 Maven dependency 引用格式**示例，例如：

   ```xml
   <dependency>
       <groupId>com.cainiao.payroll</groupId>
       <artifactId>payroll-boot-starter-util</artifactId>
       <version>${cn-payroll.starter.version}</version>
   </dependency>
   ```

4. 若版本号通过 `<properties>` 管理，同时给出 `pom.xml` 中 `<properties>` 片段示例。
5. 提醒用户：发布前确认 SNAPSHOT / RELEASE 策略，发布后通知下游项目升级。

## 习惯五：新对话启动时回顾历史 Q&A（Restore Context On Session Start）

> 本习惯是习惯三的"读取侧"——和习惯三共同构成本 Skill 的"项目记忆层"，缓解每开新会话就丢失上下文的窘况。

### 触发条件
**每个新对话的第一次涉及本项目的实质性请求**（不论是写代码、问问题、还是讨论方案）触发，且需满足：
- 当前工作目录或用户提及的目录下**存在 `.good-habits/`** 文件夹。
- 该文件夹下**至少有 1 个 Q&A 记录文件**。

> 如何判断"新对话"：本会话尚未读取过该项目的 `.good-habits/` 任何文件即视为新对话。同一会话内**只触发一次**，避免重复读档。

### 行为规范
1. 在动手回答用户问题之前，**先列出** `.good-habits/` 下的文件名（按文件名倒序，即最新优先）。
2. **读取最近 3 ~ 5 条**记录的全文（数量按用户问题的复杂度自行判断；用户问题越宽泛，读得越多）。如果用户的问题与某个旧主题强相关，再针对性地读对应文件。
3. 在回答前用一段简短的"上下文回顾"告知用户读到了什么，例如：
   > 📚 我刚扫了 `.good-habits/` 下最近 3 条记录：[2026-05-12] 重构了登录模块、[2026-05-13] 升级了 starter-util 到 1.1.0、[2026-05-14] 新增了 X 接口。下面基于这些上下文回答你的问题……
4. 若发现历史记录中存在**未完成事项 / 已知风险**，且与当前请求相关，**主动提醒**用户。
5. 若 `.good-habits/` 不存在或为空，跳过本习惯。在首次因习惯三创建 Q&A 文件时，可顺带告知用户：「以后新对话我会自动回看这个目录，避免你重复交代背景。」

### 反例（禁止行为）
- 看到 `.good-habits/` 存在却不读，直接回答——❌
- 把目录下所有文件全文塞进上下文，浪费 token——❌
- 在同一会话里反复重读相同文件——❌

### 与外部 Memory 系统的关系
- 本习惯**仅依赖项目本地的 `.good-habits/`**，不依赖任何 Anthropic memory、外部 KV、向量库等。
- 如果用户的客户端启用了平台级 memory，二者并行不悖：平台 memory 偏向"用户画像、偏好、跨项目共识"，`.good-habits/` 偏向"本项目的工作流水账"。

## 习惯六：Java 注释规范（Java Comment Standard）

### 触发条件
满足以下任一条件即触发：
- 用户要求**新建** Java **类 / 枚举 / 接口 / 方法 / 静态变量 / 实例变量**。
- 在已有 Java 文件中**新增**上述任一元素（例如给已有类新增一个方法）。

### 不触发条件
- 纯阅读、解释、重命名、移动 Java 代码。
- 修改已有元素的实现（未新增元素）。
- **对已有完整注释的元素的任何改动**——不强制重写其注释。

### 行为规范
1. **类 / 枚举 / 接口** 的 Javadoc 必须包含三行：
   - `@author 鹤童`（固定值，作者名见 `references/java-comment-standard.md`）
   - `@desc [一句话描述本类/枚举/接口的职责]`
   - `@date [当前本地时间，格式严格为 YYYY/MM/DD HH:mm]`
2. **方法** 的 Javadoc 必须包含：
   - 首行：方法描述
   - 描述与参数之间空一行
   - 每个参数一行 `@param`，按签名顺序
   - 非 void 方法必须有 `@return`
3. **静态变量、实例变量** 上方使用单行 Javadoc 描述。
4. **方法内变量** 仅在变量含义**不直观**时加行内注释（`// 描述`）；变量名已自解释（如 `userId`、`totalCount`）则**不加**。
5. 7 类元素的完整格式样例与自查清单见 `references/java-comment-standard.md`，写代码前应先查阅。

### 反例（禁止行为）
- 新建一个 Java 类却不加 Javadoc——❌
- 给方法补注释时漏写 `@param` 或 `@return`——❌
- 重写一个已有合规注释的方法的 Javadoc，仅为了刷新 `@date`——❌（不强制重写）
- 给所有方法内变量都加注释，哪怕变量名已自解释（如 `// 用户ID` 配上 `String userId`）——❌
- `@date` 用占位符 `[当前日期]` 直接提交，没有替换为真实时间——❌

### 与其他习惯的关系
- 与**习惯一**串联：用户提需求时仍要先按习惯一出方案，**方案通过后写代码时**才按习惯六的格式落注释。
- 与**习惯三**正交：写完代码后按习惯三记录 Q&A，本规范不影响 Q&A 文件内容。

## 习惯执行优先级与组合规则

> 在所有场景下，**新对话的第一次实质性请求**都先评估是否触发**习惯五**（读 `.good-habits/`），再进入下表流程。

| 场景 | 触发顺序 |
| --- | --- |
| 新对话、且项目下有 `.good-habits/` | 习惯五 → 进入下表对应行 |
| 用户提需求且涉及多种技术选型 | 习惯二 → 习惯一 → 执行 → 习惯三 |
| 用户提单一明确需求 | 习惯一 → 执行 → 习惯三 |
| 用户问二方包发布问题（不改代码） | 习惯四 |
| 用户问二方包发布问题且要改 pom.xml | 习惯四 → 习惯一 → 执行 → 习惯三 |
| 涉及 Java 文件**新建**类/枚举/接口/方法/静态变量/实例变量 | 习惯六**叠加**到上述任一行的「执行」阶段：写代码时按规范落注释 |
| 纯查看 / 解释类问题 | 不触发写动作（不进习惯一/二/三/六），但仍要评估习惯五 |

## 配套资源
- `scripts/create-qa-record.sh`：自动创建 Q&A 记录文件的 Bash 脚本。
- `scripts/create-qa-record.py`：同上，Python 版本（无 Bash 环境时使用）。
- `assets/qa-template.md`：Q&A 记录的 Markdown 模板。
- `references/plan-template.md`：习惯一的「实施方案」模板。
- `references/comparison-template.md`：习惯二的「多方案对比」模板。
- `references/maven-release-order.md`：习惯四的二方包发布顺序详细说明。
- `references/java-comment-standard.md`：习惯六的 Java 注释格式样例与自查清单。
